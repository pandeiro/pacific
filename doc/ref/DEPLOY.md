# Deployment Guide

This document describes how the Pacifica CI/CD pipeline works and how to set up the VPS.

## Architecture

The deployment uses **GitHub Actions** to build and push Docker images to **GitHub Container Registry (GHCR)**, then deploys to a VPS via SSH.

### Key Design Decisions

- **Single API Instance**: Both production and staging frontends hit the same API on port 4900
- **Environment Differentiation**: Done via static assets only (nginx serves different files based on hostname)
- **GHCR Authentication**: VPS uses a GitHub Personal Access Token to pull images
- **Security**: API binds to localhost only (127.0.0.1:4900), accessed via nginx reverse proxy

### Services

- **PostgreSQL**: Runs via `docker-compose.base.yml`, shared across deployments
- **API**: Single instance on port 4900 (localhost only)
- **Scraper**: Runs periodically via profile

### File Structure on VPS

```
/opt/pacifica/
├── docker-compose.base.yml    # Postgres service
├── docker-compose.prod.yml    # API + scraper (generated by CI)
├── migrations/                # Database migrations
└── seed/                      # Seed data

/var/www/pacifica/
├── prod/                      # Production static assets
└── staging/                   # Staging static assets
```

## GitHub Secrets Required

Set these via the GitHub CLI:

```bash
# SSH access
gh secret set DEPLOY_KEY < ~/.ssh/your_deploy_key
gh secret set DEPLOY_HOST --body "pch.onl"
gh secret set DEPLOY_USER --body "deploy"

# Deployment paths
gh secret set DEPLOY_SERVICES_PATH --body "/opt/pacifica"
gh secret set DEPLOY_STATIC_PATH_PROD --body "/var/www/pacifica/prod"
gh secret set DEPLOY_STATIC_PATH_STAGING --body "/var/www/pacifica/staging"

# Application secrets
gh secret set POSTGRES_PASSWORD --body "your_secure_password"

# GHCR authentication (create at https://github.com/settings/tokens)
# Token needs: read:packages scope
gh secret set GHCR_TOKEN --body "ghp_xxxxxxxxxxxxxxxxxxxx"
```

## VPS Setup (One-time)

1. **Create deploy user**:
```bash
sudo useradd -m -s /bin/bash deploy
sudo usermod -aG docker deploy
```

2. **Add deploy key to authorized_keys**:
```bash
sudo mkdir -p /home/deploy/.ssh
sudo echo "YOUR_PUBLIC_KEY" > /home/deploy/.ssh/authorized_keys
sudo chown -R deploy:deploy /home/deploy/.ssh
sudo chmod 700 /home/deploy/.ssh
sudo chmod 600 /home/deploy/.ssh/authorized_keys
```

3. **Create directories**:
```bash
sudo mkdir -p /opt/pacifica
sudo mkdir -p /var/www/pacifica/prod
sudo mkdir -p /var/www/pacifica/staging
sudo chown -R deploy:deploy /opt/pacifica
sudo chown -R deploy:deploy /var/www/pacifica
```

4. **Copy base docker-compose and initial files**:
```bash
# Copy from your local machine or clone the repo
cp docker-compose.base.yml /opt/pacifica/
cp -r db/migrations /opt/pacifica/
cp -r db/seed /opt/pacifica/
```

5. **Initial database setup**:
```bash
cd /opt/pacifica
docker compose -f docker-compose.base.yml up -d
# Wait for postgres to be healthy, then run migrations
docker compose -f docker-compose.base.yml exec postgres psql -U pacifica -d pacifica -f /migrations/001_tables.sql
```

## Nginx Configuration

Example nginx config for the VPS:

```nginx
# /etc/nginx/sites-available/pch.onl

# Production server
server {
    listen 443 ssl http2;
    server_name pch.onl;

    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;

    # Static assets
    location / {
        root /var/www/pacifica/prod;
        try_files $uri $uri/ /index.html;
    }

    # API proxy (same for both environments)
    location /api/ {
        proxy_pass http://127.0.0.1:4900/;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

# Staging server
server {
    listen 443 ssl http2;
    server_name staging.pch.onl;

    ssl_certificate /path/to/cert.pem;
    ssl_certificate_key /path/to/key.pem;

    location / {
        root /var/www/pacifica/staging;
        try_files $uri $uri/ /index.html;
    }

    # API proxy (same production API)
    location /api/ {
        proxy_pass http://127.0.0.1:4900/;
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection "upgrade";
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

## CI/CD Pipeline

### On Pull Request

1. Runs validation (lint, typecheck, tests)
2. Builds frontend with `VITE_API_URL=https://staging.pch.onl/api`
3. Builds Docker images with SHA tag
4. Deploys frontend to staging path only

### On Main Branch Push

1. Runs validation
2. Builds frontend with `VITE_API_URL=https://pch.onl/api`
3. Builds Docker images with `latest` tag
4. Deploys:
   - Frontend to production path
   - Docker images to VPS (logs into GHCR, pulls, restarts containers)

## Manual Deployment (if needed)

If GitHub Actions is unavailable:

```bash
# On VPS as deploy user
cd /opt/pacifica

# Authenticate to GHCR (token must be set as env var)
echo $GHCR_TOKEN | docker login ghcr.io -u USERNAME --password-stdin

# Pull and restart
docker compose -f docker-compose.base.yml -f docker-compose.prod.yml pull
docker compose -f docker-compose.base.yml -f docker-compose.prod.yml up -d

# Cleanup
docker logout ghcr.io
docker system prune -f
```

## Frontend Configuration

The frontend uses `VITE_API_URL` environment variable to determine where to make API calls:

- **Local development**: Set in `.env` as `VITE_API_URL=http://localhost:4900`
- **Production build**: Set during CI as `VITE_API_URL=https://pch.onl/api`
- **Staging build**: Set during CI as `VITE_API_URL=https://staging.pch.onl/api`

See `frontend/src/config.ts` for the API client configuration.
